# Migration Workflow

This describes the **actual, real migration history** of this repository — not the standard `prisma
migrate dev` incremental workflow most Prisma projects use. BOND OS has exactly **one** migration folder,
generated offline by diffing the schema against an empty database rather than by applying incremental
changes against a running Postgres instance, with two categories of index hand-appended afterward for
things Prisma cannot express natively. This document explains why, and exactly what was appended.

## What's actually on disk

```
packages/database/prisma/migrations/
├── migration_lock.toml
└── 20260718000000_init/
    └── migration.sql        (1,999 lines)
```

One migration, `20260718000000_init`. `migration_lock.toml` contains only:

```toml
# Please do not edit this file manually
# It should be added in your version-control system (e.g., Git)
provider = "postgresql"
```

There is no second migration, no incremental "phase 2 adds X" migration per phase — despite the schema
itself being built up additively across ten phases (P0–P9, see the header comments throughout
[`schema.prisma`](../../packages/database/prisma/schema.prisma) and [schema.md](./schema.md)), the SQL
was (re)generated as a single fresh init script each time, most recently dated 2026-07-18.

## Why offline diffing instead of `prisma migrate dev`

`prisma migrate dev` needs a **live, reachable Postgres instance** at `DATABASE_URL` for every single
migration step — it applies the migration, then introspects the result to keep the migration history
table in sync. Some environments this codebase was built and iterated in (dev sandboxes/CI-less
contexts) had no live Postgres reachable at all. Rather than blocking schema iteration on infrastructure
that wasn't always available, this project standardized on `prisma migrate diff`, which needs **no**
database connection — it computes the SQL diff between two schema *representations* (here: an empty
database and the current `schema.prisma`) purely by reading the Prisma schema AST.

`packages/database/package.json` defines this explicitly as its own script, separate from the standard
Prisma lifecycle scripts:

```json
{
  "scripts": {
    "generate": "prisma generate",
    "validate": "dotenv -e ../../.env -- prisma validate",
    "migrate:dev": "dotenv -e ../../.env -- prisma migrate dev",
    "migrate:deploy": "dotenv -e ../../.env -- prisma migrate deploy",
    "migrate:diff:init": "dotenv -e ../../.env -- prisma migrate diff --from-empty --to-schema-datamodel=prisma/schema.prisma --script",
    "studio": "dotenv -e ../../.env -- prisma studio",
    "seed": "dotenv -e ../../.env -- tsx prisma/seed.ts"
  }
}
```

`migrate:dev` and `migrate:deploy` are present (this is a real Prisma project, and a real reachable
Postgres — e.g. in production or a fully-provisioned dev environment — can and should use them going
forward for any *new* migration), but the migration folder that actually exists in this repo today was
produced by `migrate:diff:init`:

```sh
pnpm --filter @bond-os/database migrate:diff:init > packages/database/prisma/migrations/20260718000000_init/migration.sql
```

i.e. `prisma migrate diff --from-empty --to-schema-datamodel=prisma/schema.prisma --script` — "diff from
an empty database to exactly what `schema.prisma` describes right now, and print the resulting SQL
script." Every time the schema changed materially across the nine phases, this was the same command
that regenerated the single `20260718000000_init/migration.sql` from scratch, rather than appending a
new dated migration folder alongside it. That is why the repository has one migration whose SQL already
reflects the full, final, ten-domain, 67-model schema, instead of nine-plus incremental migrations that
would trace the phase-by-phase history the way the `schema.prisma` header comments do.

**Practical consequence:** this migration is only valid applied to a genuinely empty database (hence
`--from-empty`). It is not a safe "upgrade path" migration for an existing populated database that
predates it — there is no prior migration for it to diff against. Provisioning a new environment means
running `prisma migrate deploy` (or `migrate:dev` in local development) once against an empty Postgres
database, which applies this one file.

## What's inside `migration.sql`

The generated portion (Prisma's own diff output) follows the standard structure, in this order:

1. `CREATE SCHEMA IF NOT EXISTS "public"`
2. `CREATE EXTENSION IF NOT EXISTS "vector"` — required by `datasource db { extensions = [vector] }` in
   `schema.prisma`, since `Embedding.vector` is a pgvector column.
3. 46 `CREATE TYPE ... AS ENUM (...)` statements — one per Prisma `enum`.
4. ~67 `CREATE TABLE` statements — one per Prisma `model`, plus one extra hidden join table,
   `_CustomerToProject` (the implicit many-to-many between `Customer` and `Project` — see
   [relationships.md](./relationships.md)).
5. ~120 `CREATE INDEX`/`CREATE UNIQUE INDEX` statements — one per `@@index`/`@@unique` (and each
   `@unique` single-field constraint), matching every index documented per-model in
   [schema.md](./schema.md).
6. ~120 `ALTER TABLE ... ADD CONSTRAINT ... FOREIGN KEY ... ON DELETE ... ON UPDATE CASCADE` statements —
   one per real Prisma relation, with the exact `Cascade`/`SetNull`/`Restrict` behavior declared in the
   schema (see [relationships.md](./relationships.md) Part 1 for what each one means and Part 4 for the
   five `Restrict` FKs). `ON UPDATE CASCADE` is Prisma's default for every FK regardless of the
   `onDelete` choice.

None of the above needed manual intervention — Prisma's diff engine handles ordinary columns, enums,
tables, indexes, unique constraints, and FKs completely on its own.

## What had to be hand-appended, and why

Prisma's schema language has **no native representation** for two Postgres features this schema depends
on, so both had to be added to the bottom of `migration.sql` by hand after every regeneration, rather than
being expressible as `@@index` or `@@unique` in `schema.prisma` itself:

### 1. Full-text search: GIN indexes over `to_tsvector(...)` expressions

Prisma has no `tsvector` column type. Rather than working around that with an `Unsupported("tsvector")`
generated column plus a sync trigger, the codebase's actual keyword-search implementation
(`packages/database/src/repositories/search.ts`, see [../search.md](../search.md)) computes
`to_tsvector(...)` **at query time**, inline in raw `$queryRaw` SQL, against `entities.title` /
`entities.description` and `knowledge_documents."parsedText"`. A plain GIN index can't be declared in
Prisma schema syntax for an *expression* (as opposed to a real column), so the two indexes over those
exact expressions were appended manually, verbatim, at the end of `migration.sql`:

```sql
-- Full-text search expression indexes (Prisma has no tsvector column type;
-- these match the to_tsvector(...) expressions in src/repositories/search.ts
-- exactly, so Postgres can actually use them). See docs/search.md.
CREATE INDEX "entities_fts_idx" ON "entities" USING GIN (to_tsvector('english', title || ' ' || coalesce(description, '')));
CREATE INDEX "knowledge_documents_fts_idx" ON "knowledge_documents" USING GIN (to_tsvector('english', coalesce("parsedText", '')));
```

These must stay **textually identical** to the expressions `search.ts` actually issues — Postgres only
uses an expression index when the query's expression matches the indexed expression exactly (modulo
trivial normalization). `search.ts` computes the same two expressions:

```ts
to_tsvector('english', e.title || ' ' || coalesce(e.description, ''))   // entities
to_tsvector('english', coalesce(kd."parsedText", ''))                   // knowledge_documents
```

If a future schema change alters `search.ts`'s query expression without updating this hand-appended
index (or vice versa), the index silently stops being used — Postgres falls back to a sequential scan
with `to_tsvector()` computed per-row, which still returns correct results, just slower. There is no
compile-time check tying the two together; they're kept in sync by convention and by this doc.

### 2. Vector similarity: an HNSW index over `embeddings.vector`

Prisma has no pgvector *index* type either (separately from the pgvector *column* type, which
`Unsupported("vector(1536)")` does handle — see [schema.md](./schema.md)'s `Embedding` entry and
[relationships.md](./relationships.md) Part 5). An HNSW (Hierarchical Navigable Small World) index is
what makes `vectorSimilaritySearch`'s cosine-distance `<=>` queries in
`packages/database/src/repositories/embeddings.ts` (see [../vector-search.md](../vector-search.md)) fast
at scale instead of a full sequential scan computing distance against every row. It, too, was appended by
hand:

```sql
-- Vector similarity index (Prisma has no pgvector index type). See docs/vector-search.md.
CREATE INDEX "embeddings_vector_hnsw_idx" ON "embeddings" USING hnsw ("vector" vector_cosine_ops);
```

`vector_cosine_ops` is the operator class matching the `<=>` cosine-distance operator
`vectorSimilaritySearch` actually issues (as opposed to `vector_l2_ops`/`vector_ip_ops` for
Euclidean/inner-product distance, which this codebase doesn't use) — the operator class and the query
operator must agree for the index to be usable, the same textual-identity concern as the FTS indexes
above, just at the index-definition level instead of the query-expression level.

## The regeneration + hand-append cycle, concretely

Every time `schema.prisma` changed in a way that affects the generated SQL (a new model, field, index,
enum, or relation), the actual workflow used to keep `migration.sql` current was:

1. Edit `schema.prisma`.
2. Run `pnpm --filter @bond-os/database migrate:diff:init` (no live database needed) to regenerate the
   *entire* `migration.sql` from scratch, diffed from empty.
3. Re-append the two full-text-search `CREATE INDEX ... USING GIN (to_tsvector(...))` statements and the
   one `CREATE INDEX ... USING hnsw (...)` statement at the end of the freshly-regenerated file, since
   step 2 overwrites the whole file and has no way to know about them.
4. Commit the regenerated `migration.sql` as-is (single migration folder, same timestamp-named directory,
   `20260718000000_init`) rather than creating a new dated migration folder alongside it.

This is why the migration folder's date (`20260718000000`) does not correspond to any single phase in the
schema's own P0–P9 history — it is the timestamp of the most recent full regeneration, not of the schema's
actual multi-phase development timeline (which is instead recorded in `schema.prisma`'s own header
comments per model, and in the phase-numbered docs like [../data-layer.md](../data-layer.md),
[../knowledge-graph.md](../knowledge-graph.md), [../embeddings.md](../embeddings.md), etc.).

## Applying the migration for real

Once a real Postgres database (with the `vector` extension available — e.g. via `pgvector/pgvector` or a
managed Postgres that supports installing it) is reachable at `DATABASE_URL`:

```sh
pnpm --filter @bond-os/database generate        # regenerate the Prisma Client into src/generated
pnpm --filter @bond-os/database migrate:deploy   # applies the one migration, non-interactively (CI/prod)
# or, for local development against a schema you're actively iterating on:
pnpm --filter @bond-os/database migrate:dev
pnpm --filter @bond-os/database seed             # optional: pnpm db:seed — populates demo data (see below)
```

`migrate:deploy` (`prisma migrate deploy`) is the non-destructive, CI/production-safe command — it applies
any migration not yet recorded as applied and does nothing else (no schema-drift prompts, no dev-only
reset-and-reapply behavior). `migrate:dev` is interactive/dev-only and would, from this point forward,
create genuinely new incremental migration folders for any further schema change — the offline
diff-and-hand-append cycle above describes how the *existing* single migration came to be, not a
restriction on how future migrations must be produced once a real database is available.

`prisma/seed.ts` (invoked via `pnpm db:seed`, configured in `package.json`'s `"prisma": { "seed": "tsx
prisma/seed.ts" }`) is a dev-only data seed — 10 users, 8 projects, 40 tasks, 25 documents, 12 meetings, 20
customers, 40 emails, a small hand-built knowledge graph (Person/Company/Product/Event entities +
Relationships + TimelineEvents), and a handful of embedded meetings/emails via the zero-config local
embedding provider. It deliberately does **not** create login-ready `Account` rows (Better Auth owns
password hashing end-to-end) — signing up for real via `/signup` is the only way to get a working login,
per the seed script's own header comment.
